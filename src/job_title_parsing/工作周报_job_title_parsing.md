## 2. 模块总体目标

`job_title_parsing` 模块的核心目标，是将招聘岗位标题和岗位描述（JD）映射到《中国职业分类大典》职业条目，输出可解释的候选结果与评分依据。当前方案采用的是 **规则 + 检索 + 融合打分** 的实现路径，而不是大模型推理路径。

当前模块已经形成较完整的端到端链路，覆盖：

- 职业大典数据预处理
- 岗位标题清洗
- 岗位描述解析
- 结构化特征抽取
- 双路 BM25 检索
- 字符 n-gram 补充召回
- 层级过滤
- 多信号融合打分
- 批量匹配与并发执行
- 效果评估与 CLI 运行入口

## 2026-03-24：建立岗位名称解析主干能力

### 变更文件

- `occupation_dict_manager.py`
- `occupation_parser.py`
- `alias_builder.py`
- `update_dict.py`

### 主要工作内容

#### 1. 建立职业词典管理能力

`occupation_dict_manager.py` 的职责非常关键，它将多个数据源统一到解析器可消费的词典能力中，主要包括：

- 从 DuckDB 固定表加载职业核心词
- 从《中国职业大典》表加载类别层级映射
- 从 `dicts` 目录加载修饰词、地点词、福利词、无效后缀等辅助词典
- 输出核心词优先级、类别映射、词典统计等能力

这一步相当于把“岗位解析”从硬编码脚本，升级成了依赖外部数据源的可维护组件。

#### 2. 实现岗位名称解析器

`occupation_parser.py` 在这一阶段完成了岗位标题解析主逻辑，关键能力包括：

- 标题预清洗
  - 去括号内容
  - 去年份
  - 去编号
  - 去福利词
  - 去地点尾巴
  - 统一分隔符
- 最长后缀匹配
  - 优先匹配职业核心词，最大化识别职业主体
- 包含匹配兜底
  - 处理核心词位于标题中部的情况
- 修饰词抽取
  - 将技能、资深度、方向等信息保留为 modifiers
- 批量解析与 DuckDB 表级解析
  - 说明该模块已考虑真实数据集批处理场景

#### 3. 建立职业别名机制

`alias_builder.py` 引入了两类别名能力：

- 基于后缀裁剪的弱别名生成
- 基于人工维护映射的口语岗位名规范化

例如，“导购”“骑手”“配送员”等口语表达可以被映射到更规范的职业名称。这一步对招聘数据尤其重要，因为岗位命名往往高度非标准化。

#### 4. 基于评估结果修复核心词词典

`update_dict.py` 直接体现了“评估驱动词典修订”的工作方式，具体做了：

- 新增 `工艺工程师`、`品质工程师`、`质量工程师`
- 新增 `市场营销经理`
- 调整 `售后客服`、`在线客服`、`电话客服` 优先级
- 移除重复或分类不合理的核心词项

### 技术价值

- 完成了“词典管理 -> 标题清洗 -> 核心词识别 -> 修饰词抽取”的基础闭环
- 明确了招聘岗位解析采用“规则 + 词典 + 优先级”的方法论
- 为后续职位匹配提供了标准化输入

### 阶段产出评价

这一阶段可以视为 `job_title_parsing` 模块的第一版可用内核。虽然当时尚未形成完整的职业召回和打分体系，但已经把标题解析这条关键前置链路打通。

## 2026-04-11：完成职业匹配系统的生产化集成

### 变更文件

- `catalog_preprocessor.py`
- `title_cleaner.py`
- `jd_parser.py`
- `feature_extractor.py`
- `bm25_index.py`
- `ngram_retrieval.py`

- `cli.py`

- `README.md`
- `__init__.py`
- `alias_dict_example.json`

### 主要工作内容

#### 1. 建立职业大典预处理层

`catalog_preprocessor.py` 完成了职业大典数据的结构化清洗与标准化，主要包括：

- 统一职业大典字段
- 切分 `tasks`
- 构建 `task_list`、`task_text_joined`
- 生成 `title_clean`、`desc_clean`
- 构建 `hierarchy_text`
- 基于 `AliasBuilder` 生成 `aliases`
- 为检索建立：
  - `retrieval_title_text`
  - `retrieval_desc_text`
  - `retrieval_task_text`

这一步的价值在于：不是直接拿原始大典字段做检索，而是先加工成更适合召回与匹配的文本字段。

#### 2. 建立岗位标题去噪模块

`title_cleaner.py` 主要解决招聘平台岗位标题中的噪音问题，包括：

- 营销词
- 福利词
- 薪资表达
- 门店/地点尾巴
- 编号/型号尾缀
- 平台品牌前缀

这说明团队已经非常清楚招聘标题的真实数据问题，不再把岗位标题当作规范文本处理，而是面向“脏数据环境”进行工程设计。

#### 3. 建立 JD 解析与结构化特征抽取

`jd_parser.py` 和 `feature_extractor.py` 在这一阶段承担了从岗位描述中提取有效信号的职责。

`jd_parser.py` 主要输出：

- `jd_clean`
- `jd_sentences`
- `core_task_sentences`
- `domain_keywords`

`feature_extractor.py` 主要输出：

- `platform_terms`
- `domain_terms`
- `function_terms`
- `object_terms`
- `conflict_terms`
- `title_terms`
- `jd_terms`

这说明系统已经从“只看标题”升级为“标题 + JD + 结构化业务词”的多信号匹配模式。

#### 4. 建立双路检索能力

`bm25_index.py` 和 `ngram_retrieval.py` 共同构成候选召回层。

其中：

- `bm25_index.py` 提供轻量本地 BM25 实现
- 支持 `search_title` 与 `search_tasks` 双通道检索
- `ngram_retrieval.py` 通过字符 2-gram / 3-gram overlap 补充分词误差场景

这类设计非常适合中文招聘文本，因为岗位名往往较短，分词误差会显著影响召回效果，字符 n-gram 可以作为稳定补充信号。

#### 5. 建立层级过滤能力

`hierarchy_filter.py` 和 `hierarchy_keyword_builder.py` 引入了“职业大类”层面的弱约束：

- 通过关键词推断岗位所属大类
- 在候选集中过滤或加分匹配的大类
- 支持从职业大典自动构建层级关键词词典

这一步的意义在于：即使标题本身较泛，也可以通过行业词、动作词、领域词等信号，对候选职业范围进行约束，减少跨大类误匹配。

#### 6. 建立融合打分机制

`scoring.py` 将不同来源的信号统一为最终分数，当前权重配置为：

- `title_weight = 0.45`
- `task_weight = 0.30`
- `desc_weight = 0.15`
- `hierarchy_weight = 0.10`
- `alias_bonus = 0.10`
- `title_direct_bonus = 0.18`
- `task_overlap_bonus = 0.08`
- `generic_title_penalty = 0.08`
- `conflict_penalty = 0.12`

除了基础加权外，还加入了：

- alias 精确命中奖励
- title 直接命中奖励
- task overlap 奖励
- 泛岗位名惩罚
- 领域冲突惩罚
- 置信度与风险标记构建

这说明系统已经不仅仅输出一个分数，而是开始具备“风险识别”和“人工复核提示”能力。

#### 7. 建立主匹配流水线

`matching_pipeline.py` 是整个模块的中枢，串联了以下能力：

- 岗位标题清洗
- JD 解析
- 结构化特征抽取
- 层级过滤
- BM25 检索
- n-gram 相似度补充分数
- 分数融合
- Top-K 候选构建
- 并发批处理

同时还支持：

- 进程池 worker 初始化
- 分片匹配
- DataFrame 批量处理
- 与 DuckDB 预处理结果联动

这标志着模块已经从单点算法脚本升级成可运行的工程流水线。

#### 8. 建立评估与命令行入口

`matching_evaluator.py`、`evaluate_matching.py` 与 `cli.py` 进一步补齐了使用闭环：

- 支持 `top1 accuracy`
- 支持 `top3 recall`
- 支持 `top5 recall`
- 支持 `unmatched rate`
- 支持并行评估
- 支持 DuckDB / CSV 双路径运行
- 支持 `preprocess-catalog`、`match`、`evaluate` 等命令

这一步非常重要，因为它让模块具备了：

- 可执行性
- 可评估性
- 可批量运行性
- 可在数据仓库环境中复用的能力

### 阶段产出评价

截至 2026-04-11，`job_title_parsing` 已不再是单纯的“岗位解析脚本集合”，而是已经形成一个具备明确输入输出、可落库、可批处理、可评估、可扩展的职业匹配子系统。

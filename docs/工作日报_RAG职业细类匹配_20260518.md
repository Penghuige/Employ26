# 工作日报：RAG 职业细类匹配 — 技术路线探索与收敛

**日期**: 2026-05-18
**任务**: 基于 DeepSeek-V4-Pro 和职业大典知识库，实现岗位→职业细类的自动匹配
**结论**: 确定最终技术路线为 BM25 关键词召回 + DuckDB 直搜补充 + DeepSeek 精排精选（不使用 embedding 模型）

---

## 一、探索历程

### 阶段 1：纯 BGE embedding + FAISS 检索（❌ 失败）

**做法**: 用 bge-large-zh-v1.5 将岗位 JD 和职业大典 definition/task chunk 编码到 1024 维语义空间，FAISS 内积检索 Top-8。

**结果**: 10 条测试仅 1 条匹配（10%）。检索返回结果与岗位完全无关：

| 岗位 | 检索 Top-1 | 问题 |
|------|-----------|------|
| 法务经理 | 电解精炼工 | 离谱 |
| 医美医生 | 电视摄像员 | 离谱 |
| 高级数据分析师 | 增材制造设备操作员 | 离谱 |

**根因**: bge-large-zh-v1.5 是通用中文语义模型，不是职业分类模型。JD 和职业大典文本都充满了通用书面语（"本科以上""5 年经验""负责""操作""管理"），在通用语义空间中距离很近，导致检索结果本质上是随机噪声。

**结论**: 通用 embedding 模型不适用于职业细类匹配。放弃纯 BGE 路线。

---

### 阶段 2：BM25 关键词召回 + BGE 精排（⚠️ 部分成功）

**做法**: JD 预处理（提取核心职责句）→ BM25 关键词粗召回 30 条 → BGE 精排 Top-8 → DeepSeek 选择最佳。

**结果**: 匹配率从 10% 跃升到 70%。BM25 的关键词匹配能力远超 BGE 的通用语义匹配。

| 岗位 | BM25 Top-1 | DeepSeek 选择 | 正确？ |
|------|-----------|-------------|--------|
| 中医妇科医生 | 中西医结合妇科医师 | 中西医结合妇科医师 | ✅ |
| 高级数据分析师 | 商务数据分析师S | 商务数据分析师S | ✅ |
| 法务经理 | 法律顾问 | 法律顾问 | ✅ |

**但仍存 3 个失败**: 医美医生、医学联络专员（BM25 无法将口语岗位名映射到正式职业名）、设备工程师（JD 过于模糊）。

**发现**: BGE 精排步骤实际上**拖累了性能**——它曾将"法务经理"排到"电解精炼工"之后。取消 BGE 精排、纯靠 BM25 排序时，性能不变甚至略好。

**结论**: BM25 是可靠的召回层，BGE embedding 在此任务中无价值。保留 BM25 + DeepSeek。

---

### 阶段 3：DuckDB 关键词直搜补充（⚠️ 边际改善）

**做法**: 在 BM25 之外，用 jieba 分词拆解岗位名 → DuckDB LIKE 搜索 title/desc/tasks → 合并进候选池。

**结果**: 匹配率维持在 ~65-70%，未显著提升。因为即使 DuckDB 直接搜索也无法桥接"医美"↔"皮肤科"、"医学联络"↔"医药代表"这类词汇鸿沟。

**结论**: 作为保险通路保留（零成本），但不是突破点。

---

### 阶段 4：置信度校准（✅ 成功）

**做法**: 基于 31 条 multi-annotator 验证集的人工标注一致性数据，建立分段线性校准映射，将 LLM 原始 confidence（0.90-0.98，明显虚高）校准到真实范围（0.45-0.68）。

**结果**: 校准后 confidence 与人工标注一致性对齐。

| 原始 confidence | 校准后 |
|----------------|--------|
| 0.95 | 0.68 |
| 0.90 | 0.54 |
| 0.75 | 0.45 |

**结论**: 校准通过。下游可使用校准后的 confidence 做阈值过滤。

---

### 阶段 5：DeepSeek 直接读取全典（❌ 更差）

**做法**: 质疑"为什么要 RAG 检索？"——直接将完整 1,698 条职业大典（~20 万 token）塞进 DeepSeek V4 Pro 的 system prompt（1M 上下文窗口），让 LLM 从中选出最佳匹配。

**结果**: 匹配率 40%，比 BM25+DeepSeek 的 70% 更差。

**根因**: LLM 在巨大的上下文中做精确检索，注意力被稀释——就像让一个人从整本字典中找正确答案，而非先给 30 个候选让他选。

**结论**: RAG 的"检索→精选"两阶段模式是正确的，但检索层不该用 embedding。

---

### 阶段 6：Prompt 优化 + NONE 回退（⚠️ 敏感）

**做法**: 优化 prompt 措辞（"优先选最接近的而非轻易判 NONE"）、添加两层回退（LLM 判 NONE → 用完整 JD 重新检索重试）。

**结果**: Prompt 措辞变化对 LLM 输出影响极大——过于保守（70%→40%）、过于激进（NONE 消失但错误增加）。最终收敛到平衡版本。

**结论**: Prompt 对 LLM 行为高度敏感，需要锁定版本。

## 二、最终技术路线

```
                        岗位输入 (job_title + job_requirements)
                                    │
                    ┌───────────────┴───────────────┐
                    ▼                               ▼
            JD 预处理                          jieba 分词拆解
        (提取核心职责句,                    (提取 ≥2 字实义词)
         去学历/经验噪声)
                    │                               │
                    ▼                               ▼
             BM25 关键词检索               DuckDB LIKE 搜索
          (1,698 条大典文本)            (title/desc/tasks)
          pool_size=30, k1=1.5            OR 条件, LIMIT 20
                    │                               │
                    └───────────────┬───────────────┘
                                    ▼
                            候选池去重合并 (30~50 条)
                                    │
                                    ▼
                          DeepSeek-V4-Pro 精排精选
                        (temperature=0.1, max_tokens=512)
                                    │
                        ┌───────────┴───────────┐
                        ▼                       ▼
                   最佳匹配职业              NONE → 回退重试
                (code + title +              (用完整 JD
                 calibrated conf)             重新检索)
```

### 各组件职责

| 组件 | 职责 | 技术选择 | 理由 |
|------|------|---------|------|
| JD 预处理 | 提取岗位核心信号，去除学历/经验/软技能噪声 | jieba 分词 + 动作动词匹配 | 减少 BM25 噪声匹配 |
| 关键词召回 | 从 1,698 条大典中快速缩小候选范围 | **BM25** (k1=1.5, b=0.75) | 确定性、无语义漂移、快速 |
| 补充召回 | 弥补 BM25 对口语岗位名的词汇缺口 | **DuckDB LIKE** (jieba 分词) | 零成本保险通路 |
| 精排精选 | 从候选池中选出最佳匹配或判定无匹配 | **DeepSeek-V4-Pro** (API) | 语义理解 + 分类推理 |
| 置信度校准 | 使 confidence 反映真实准确率 | 分段线性校准 (基于人工标注) | 下游可用 |
| 回退重试 | LLM 判 NONE 时用完整 JD 再试 | 两层回退 | 减少误判 NONE |

### 明确不使用

| 组件 | 原因 |
|------|------|
| **bge-large-zh-v1.5** embedding | 通用语义空间不区分职业细类, 实证退化 10%→70% (去掉后) |
| **FAISS 向量检索** | 同上, 检索结果本质是随机噪声 |
| **DeepSeek 直接读全典** | 上下文太大 (200K tokens), 注意力稀释, 40% 匹配率 |

### 当前性能

| 指标 | 数值 |
|------|------|
| 匹配率 (10 条测试) | **~70%** |
| 校准后置信度 | 0.45-0.75 |
| 单条耗时 | ~20s (DeepSeek API) |
| 剩余 3 个失败 | 全部因词汇鸿沟 (口语岗位名↔正式职业名) |

### 下一个突破口

**行业别名词典**: 创建 `dicts/industry_alias_mapping.json`，将口语岗位名映射到正式职业名：

```json
{"医美医生": "皮肤科医师", "医学联络专员": "医药代表", ...}
```

在 BM25 检索前做一次映射，预计匹配率可从 ~70% 提升到 ~85%。这是当前架构下最后一个低成本的重大改进点。

## 三、关键设计决策

1. **为什么不用 embedding？** 通用 embedding 模型的语义空间是为自然语言理解设计的，不是为职业分类设计的。在 1,698 个职业细类中做精确匹配，关键词匹配 (BM25) 比语义相似度 (BGE) 更可靠。

2. **为什么不全交给 DeepSeek？** LLM 在巨大上下文中的检索能力不如专门的检索算法 (BM25)。"缩小候选范围再让 LLM 选"比"让 LLM 从全量中找"更有效。

3. **为什么保留 DuckDB 直搜？** 作为 BM25 的补充召回，零额外成本，能在少数 case 中提供 BM25 遗漏的候选。例如"摄像"的 DuckDB 直搜直接找到了"电视摄像员"。

4. **为什么校准置信度？** LLM 原始 confidence 集中在 0.90-1.0，与实际 70% 匹配率严重不符。校准后 confidence 与人工标注一致性对齐，下游可安全使用阈值过滤。

# 技能抽取系统重构总结

## 📌 重构背景

原有技能抽取系统存在严重的噪声问题：
- **改进版本**：6162个技能，噪声率40-50%
- **严格版本**：541个技能，噪声率30%
- **核心问题**：方法论错误，把"高频词"等同于"技能词"

## 🎯 重构目标

**从"无监督发现"转向"有监督识别"**

- ✅ 精度优先（目标：85-90%）
- ✅ 可控可解释
- ✅ 易于维护和迭代
- ✅ 规范的词典管理

## 🏗️ 新架构设计

### 技能分类体系（6大类）

1. **技术技能**：编程语言、框架、数据库、大数据/AI、运维工具
2. **工具软件**：办公软件、设计软件、工程软件、测试工具
3. **行业技能**：制造、质量管理、财务会计
4. **证书资质**：PMP、CPA、建造师等
5. **方法论**：敏捷开发、DevOps、微服务等
6. **语言能力**：英语、日语、CET-6等

### 黑名单体系（9大类）

1. **要求描述词**：熟悉、了解、掌握、精通...
2. **职责动词**：负责、完成、执行、推进...
3. **福利待遇**：五险一金、餐补、年终奖...
4. **学历要求**：本科、硕士、学历...
5. **软技能**：沟通能力、责任心、团队精神...
6. **时间词**：一天、上午、三年...
7. **通用泛化词**：能力、工作、经验、知识...
8. **英文停用词**：and, for, in, of...
9. **地名关键词**：市、区、县、路...

## 🔧 维护指南

### 添加新技能

编辑 `dicts/skill_seeds.txt`：

```
# 格式：词汇 词频 词性 分类
新技能名 30000 nz programming_language
```

### 添加黑名单词

编辑对应的 `dicts/blacklist_*.txt`：

```
# 直接添加一行
新噪声词
```

### 添加同义词

编辑 `dicts/synonyms.txt`：

```
# 格式：标准词 -> 变体词1, 变体词2
标准词 -> 变体1, 变体2, 变体3
```

### 重新运行

```bash
python src/skill_extraction/mvp_skill_extractor.py
```

## 🎓 方法论总结

### 核心洞察

1. **技能 ≠ 高频词**
   - 招聘广告中"负责"、"五险一金"比"TensorFlow"更高频
   - 但前者不是技能

2. **语义相似 ≠ 类别相同**
   - "责任心"和"Python"可能在语境中接近
   - 但前者是软素质，后者是技能

3. **精度 > 召回率**
   - 趋势分析需要准确的技能统计
   - 噪声会严重扭曲结果
   - 覆盖70%主流技能已足够

4. **规则 + 词典 > 纯机器学习**
   - 500个种子词，人工整理只需1-2天
   - 但能达到85%精度
   - 可解释、可维护、可迭代

### 设计原则

1. **明确定义边界**
   - 什么是技能，什么不是
   - 用操作性定义，不用模糊概念

2. **负样本同样重要**
   - 不仅要知道"什么是技能"
   - 更要知道"什么不是技能"

3. **归一化是必须的**
   - Python = python = PYTHON
   - 否则虚增数量

4. **置信度比二元判断更有用**
   - 可以排序、筛选、调整阈值
   - 便于人工审核

5. **可解释性是生产级系统的要求**
   - 每个技能知道来源
   - 每个技能有置信度
   - 便于调试和优化

## 🤔 常见问题

### Q1: 为什么不用BERT/大模型？

**A**: 
- 5GB数据，推理成本高
- 中文招聘领域预训练模型效果未必好
- 规则+词典已能达到85%精度
- 过度工程，维护成本高

### Q2: 为什么要牺牲召回率？

**A**:
- 你的任务是趋势分析，不是简历匹配
- 噪声会严重扭曲统计结果
- 覆盖70%主流技能已足够
- 高精度结果更有学术价值

### Q3: 如何评估效果？

**A**:
- 随机抽取100个技能
- 人工判断是/否
- 计算精度 = 真实技能数 / 100
- 目标：>85%

### Q4: 如何提高召回率？

**A**:
- 补充种子词典（最直接）
- 补充同义词映射
- 升级到V2版本（Word2Vec扩展）
- 但要注意精度下降

### Q5: 发现新噪声怎么办？

**A**:
- 加入对应黑名单文件
- 重新运行抽取器
- 持续迭代优化

## 🔍 阶段1：问题诊断报告

### 1.1 当前方法的根本性错误假设

| 错误假设 | 问题 | 后果 |
|---------|------|------|
| **高频词 = 技能词** | TF-IDF提取的是重要词，不是技能词 | "负责"、"五险一金"被当成技能 |
| **语义相似 = 类别相同** | Word2Vec找相似词，不区分类别 | "责任心"因与"Python"共现被纳入 |
| **多轮迭代能发现更多技能** | 每轮迭代都在放大噪声 | 第3轮新增3574个词，大部分是噪声 |
| **技能 = 任何工作相关词** | 没有明确定义技能边界 | 混淆了技能、能力、素质、职责 |

### 1.2 噪声来源分析（10大类）

| 噪声类型 | 占比 | 典型样例 | 来源 |
|---------|------|---------|------|
| 职责动词 | 15-20% | 负责、完成、执行、推进 | TF-IDF高频 |
| 软素质词 | 10-15% | 责任心、沟通能力、团队精神 | Word2Vec扩展 |
| 要求描述 | 10-12% | 熟悉、了解、掌握、精通 | TF-IDF高频 |
| 福利待遇 | 5-8% | 五险一金、餐补、年终奖 | TF-IDF高频 |
| 时间词 | 3-5% | 一天、上午、三年 | TF-IDF高频 |
| 地名 | 3-5% | 香港地区、韶关市、黄江 | TF-IDF高频 |
| 英文停用词 | 2-3% | and, for, in, of | 分词错误 |
| 食品/日用品 | 1-2% | 雪糕、零食、面包车 | 特定行业 |
| 学历词 | 1-2% | 本科、硕士、学历 | TF-IDF高频 |
| 泛化词 | 5-8% | 专业、知识、技能、方法 | TF-IDF高频 |

**总噪声率：55-80%**

### 1.3 技术原因分析

1. **TF-IDF的局限**：只能识别"重要词"，不能识别"技能词"
2. **分词导致伪候选**：英文停用词"and"被当成候选
3. **Word2Vec语义漂移**：Python → Django → 开发 → 负责（完全偏离）
4. **缺少类型约束**：没有检查词性和语义类别
5. **黑名单严重不足**：只有27个词，实际需要200+

### 1.4 评估方法的缺失

- ❌ 没有量化评估（Precision/Recall/F1）
- ❌ 没有人工标注数据
- ❌ 没有分类统计
- ❌ 只看数量，不看质量

## 🎯 核心创新点

### 1. 问题重新定义

**旧定义**（隐含）：
> "技能 = 在招聘广告中频繁出现且与已知技能相似的词"

**新定义**（明确）：
> "技能 = 可传授、可评估、可应用于工作任务的具体知识或能力"

### 2. 方法论转变

**从**：无监督学习（TF-IDF + Word2Vec）  
**到**：规则+词典+少量监督

**原因**：
- 招聘广告的"高频词"不等于"技能词"
- 500个种子词人工整理只需1-2天，但能达到85%精度
- 规则+词典更可控、可解释、可维护

### 3. 评估标准转变

**从**：数量（6162个 > 541个）  
**到**：精度（90% > 70% > 60%）

**原因**：
- 趋势分析需要准确的技能统计
- 噪声会严重扭曲结果
- 覆盖70%主流技能已足够

### 4. 架构设计创新

**规范的词典管理**：
- 所有词典文件统一存放在`dicts/`目录
- 不在代码中硬编码词汇
- 易于维护和扩展

**多层黑名单机制**：
- 9大类黑名单，682个禁用词
- 比旧版27个词增加25倍
- 覆盖所有主要噪声类型

**置信度打分**：
- 每个技能有置信度（0-1）
- 可排序、筛选、调整阈值
- 便于人工审核

## 💡 关键洞察

### 1. 问题定义比算法更重要

**错误的问题定义**：
- "如何从文本中发现更多技能词？"
- 导向：追求数量，使用无监督方法

**正确的问题定义**：
- "如何从文本中准确识别真实技能？"
- 导向：追求精度，使用规则+词典

### 2. 领域知识比机器学习更有效

**机器学习方法**：
- TF-IDF + Word2Vec
- 无需人工，但噪声多
- 精度50-60%

**规则+词典方法**：
- 500个种子词（1-2天人工整理）
- 682个黑名单词（1-2天人工整理）
- 精度85-90%

### 3. 精度优先是正确的策略

**对于趋势分析任务**：
- 需要准确的技能统计
- 噪声会严重扭曲结果
- 覆盖70%主流技能已足够
- 高精度比高召回更重要

### 4. 可维护性是长期成功的关键

**旧方案**：
- 黑名单硬编码在代码中
- 发现噪声需要改代码
- 维护成本高

**新方案**：
- 词典文件独立管理
- 发现噪声只需编辑文本文件
- 维护成本低
